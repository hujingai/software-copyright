# Software Copyright Generator（软件著作权申请材料生成器）

本 skill 从真实项目生成和复核中国软件著作权申请材料。核心目标：**先理解项目** → **确认口径** → **草稿审阅** → **正式生成** → **最终复核**，让资料可追溯、可确认、可复核。

## 基本原则

- 输出目录默认使用当前工作目录下的 `软著申请资料/`。
- 用户提出生成软著材料时，**先输出前置采集表并等待用户填写**；表单按八组维度收集：登记权属、开发发表、项目形态、业务手册、源码材料、运行截图、模板输出、正式检查。
- 所有操作优先按显式节点执行；可完整跑流程，也可单点重跑。
- **先生成 Markdown 草稿，用户确认后再生成正式 Word/PDF**。
- 正式资料写入 `软著申请资料/正式资料/`。
- 代码材料必须来自真实源码，禁止编造代码。
- 软件全称和版本号必须在申请表、用户手册、源代码、文件名、页眉页脚中保持一致。

## 确认模式

前置采集完成后，询问用户选择确认模式：

- **`interactive`（默认）**：逐项确认，每个关键节点停住等待用户输入。
- **`auto`**：用户授权后按默认值直走；登记数据类内容（申请表字段等）仍须用户确认。

## 强制门禁

以下阶段必须停住等待用户确认后才能继续：

| 阶段 | 说明 | 触发时机 |
|------|------|----------|
| `preflight` | 前置采集表填写完成 | 开始任何生成前 |
| `project` | 选定目标项目目录 | 多项目候选时 |
| `business` | 业务理解草稿确认 | 项目分析后 |
| `code-selection` | 源码文件选择确认 | 源码抽取前 |
| `screenshot-method` | 截图方式选择 | 手册生成前 |
| `markdown` | 全部草稿审阅确认 | 生成正式文件前 |
| `review` | 最终三件套复核 | 交付前 |

## 完整工作流

### 第1步：前置采集
向用户展示采集表，包含以下八组字段：

**① 登记与权属**：软件全称✅、软件简称、版本号✅、著作权人✅、著作权人类型、权利取得方式、权利范围、开发方式

**② 开发与发表**：开发完成日期✅、发表状态✅、首次发表日期、首次发表城市、软件分类、硬件环境、操作系统、运行支撑环境、开发工具、编程语言

**③ 项目形态与仓库**：仓库模式✅（单体/前后分离/多仓库）、项目根目录、前端代码位置、后端代码位置、移动端代码位置、需排除的目录

**④ 业务范围与手册**：业务领域、目标用户、申报覆盖模块、不纳入模块、核心操作流程、手册章节偏好

**⑤ 源码材料**：源码抽取策略（自动建议/用户指定）、优先纳入模块、源码排列顺序、每页行数（默认50行）、页数口径（前30+后30）、排除目录

**⑥ 运行与截图**：截图方式（用户提供/自动爬取真实截图/AI生成/暂不截图）✅、需要截图的页面、用户截图目录

**⑦ 模板与输出**：输出目录（默认 `软著申请资料/`）、输出格式（Markdown/DOCX/PDF）、有无历史申请表模板

**⑧ 正式检查**：不应出现的禁用词、一致检查重点

用户可用 `scripts/generate_input_form.py` 生成采集表 Markdown 供填写。

### 第2步：环境检查
确认运行环境支持 Markdown、DOCX/PDF 生成能力。如缺少依赖（如 `reportlab`、`python-docx`），提示用户安装。

### 第3步：定位并分析项目
按采集表中的项目根目录定位项目。若多个候选目录则询问用户。排除 `node_modules`、`dist`、`.git`、构建产物等目录。

分析内容：项目名称、编程语言、框架、文件统计、包信息、路由结构、README 摘要。

可运行 `scripts/analyze_project.py --project <路径>` 辅助分析。

### 第4步：业务理解
基于项目 README、PRD、路由、页面和源码，生成业务理解摘要：
- 软件是什么、解决什么问题
- 目标用户是谁
- 核心功能模块有哪些
- 主要操作流程

输出 `草稿/业务理解.md` 并等待用户确认。

### 第5步：登记字段确认
确认申请表所需字段：软件全称、版本号、著作权人、开发完成日期、发表状态、硬件环境、软件环境、开发工具、编程语言、软件分类、技术特点。

### 第6步：源码选择与抽取
1. 生成候选源码文件清单（按模块、按前后端分类）
2. 标注行数、语言、所属模块
3. 用户确认或调整选择范围
4. 运行 `scripts/generate_source_doc.py` 提取源代码
5. 生成 `正式资料/[软件全称]_源代码文档_V[版本号].pdf`

**源码规则：**
- 每页 50~60 行代码，A4 纸
- 超 60 页取前 30 页 + 后 30 页；不足 60 页全交
- 排除构建产物、依赖、测试固件、第三方库
- 保留正常缩进；清除密钥、token、密码
- 前后端都涉及申报时，前端放入口/路由/页面/组件，后端放 Controller/Service/Model/Entity
- 页眉必须包含：软件全称 + 版本号 + "源代码文档"
- 页码格式："第 X 页 / 共 Y 页"

### 第7步：用户手册草稿
生成《用户使用说明书》Markdown 草稿。

**手册结构：**
1. 封面（软件名称、版本号、生成日期、著作权人）
2. 引言（软件概述、适用对象、术语说明）
3. 系统要求（硬件、操作系统、依赖软件）
4. 安装与启动（安装步骤、启动方式、首次登录）
5. 功能操作说明（按菜单/页面逐一介绍）
   - 每个功能：入口位置 → 页面布局 → 操作步骤 → 结果反馈
   - 配截图预留位：`【请在此插入 XX 页面截图】`
6. 常见问题 FAQ
7. 联系方式

**手册规范：**
- A4，标题宋体16pt加粗，正文宋体12pt，1.5倍行距
- 不使用"进入方式/页面内容/操作步骤"模板堆砌
- 不写营销口号和 AI 味排比
- 截图优先来自真实系统运行截图（用户提供或自动爬取），次选 AI 生成，不截图时需有清晰预留位

### 第8步：截图处理（按用户选择）

#### 方式一：用户提供截图
- 用户将截图文件放入指定目录（如 `screenshots/`），文件名尽量匹配功能模块名
- 支持格式：PNG、JPG、JPEG、GIF、BMP、WebP
- 命名建议：`登录.png`、`温度监测.png`、`温度监测_新增.png` 等
- 生成器会根据模块名自动匹配截图文件

#### 方式二：自动爬取真实页面截图（适用于静态 Web 项目）
- 适用于纯前端静态 HTML 项目（无需后端运行）
- **操作流程**：
  1. 在源码项目根目录启动本地 HTTP 服务器：
     ```bash
     # Python 3
     python -m http.server 8080 --directory <源码目录>
     ```
  2. 确认服务器启动后，加载 `playwright-cli` 或 `agent-browser` skill 进行浏览器自动化
  3. 依次打开各功能页面的 URL（如 `http://localhost:8080/首页.html`）并截图
  4. 截图保存到 `screenshots/` 目录，按模块名命名（如 `登录.png`、`首页.png`）
- **Playwright 截图命令示例**：
  ```bash
  # 打开页面并全页截图
  playwright-cli open http://localhost:8080/首页.html --screenshot screenshots/首页.png --width 1440 --height 900
  ```
- **Agent-Browser 截图命令示例**：
  ```bash
  agent-browser snapshot --url http://localhost:8080/首页.html --output screenshots/首页.png --width 1440
  ```
- **优势**：截图内容来自真实源码渲染效果，比 AI 生成更准确可靠
- **注意事项**：
  - 仅适用于纯静态页面（HTML/CSS/JS），前后端分离项目需先确保前端可独立运行
  - 截图前确认页面已完整加载（等待 2~3 秒）
  - 需关闭浏览器弹窗、Cookie 同意栏等遮挡元素
  - 若页面依赖后端 API 数据，列表中可能显示空状态，应在手册中注明

#### 方式三：AI 生成截图（无源码或无浏览器环境时）
- 使用 `image_gen` 工具生成模拟系统界面截图
- 每个模块生成对应的操作页面截图（列表页、新增/编辑表单页、详情页）
- 截图内容应接近真实系统界面风格（数据表格、表单、图表等）
- 生成截图保存到 `screenshots/` 目录，命名规则与模块名对应
- 建议生成以下核心截图：
  - 登录页面（用户名密码表单、验证码、登录按钮）
  - 主界面/仪表盘（导航菜单、数据概览卡片）
  - 各功能模块：列表页（含数据表格、搜索栏）+ 新增/编辑页（含表单）
  - 系统设置页面（如有）

#### 方式四：暂不截图
- 手册保留清晰可见的截图预留位：【请在此插入 XX 页面截图】
- 标注图片位置、说明和图号，便于后续手动补充

#### 截图格式要求
- 推荐分辨率：1280×800 或 1440×900
- 格式：PNG（推荐，清晰无损）
- 居中放置，附题注（10pt，居中）
- 避免包含个人隐私信息

### 第9步：Markdown 确认
展示全部草稿（业务理解、申请表信息、用户手册、源码材料），等待用户全面审阅确认。

### 第10步：正式生成
一次性生成三件套正式文件到 `软著申请资料/正式资料/`，同时生成 DOCX 和 PDF 双格式：

### 源代码文档
- `[软件全称]_源代码文档_V[版本号].pdf` — PDF 版（A4，等宽字体）
- `[软件全称]_源代码文档_V[版本号].docx` — Word 版（可编辑）

### 用户手册
- `[软件全称]_用户手册_V[版本号].pdf` — PDF 版（封面 + 正文）
- `[软件全称]_用户手册_V[版本号].docx` — Word 版（含嵌入截图）

### 申请表
- `[软件全称]_申请表_V[版本号].pdf` — PDF 版（表格形式）
- `[软件全称]_申请表_V[版本号].docx` — Word 版（表格形式，可编辑）

### 截图处理
- **用户提供截图**：将截图文件放入指定目录，文件名尽量匹配功能模块名（如 `登录.png`、`温度监测.png`），生成器会自动匹配
- **无截图**：生成器在对应位置插入占位提示文字【请在此插入 XX 页面截图】
- **AI 生成截图**：可在第 8 步选择使用 image_gen 工具生成模拟系统界面截图，覆盖登录页和各功能页面

### 第11步：最终复核
对照 `references/final_review.md` 执行最终三件套复核：
- ✅ 三件套文件存在，文件名含软件全称+版本号
- ✅ 三份文档中名称、版本号、著作权人一致
- ✅ 无禁用词残留
- ✅ 用户手册截图来自真实来源或有预留位
- ✅ 源码可回溯真实项目，页数/行数/页码满足要求
- ✅ 申请表字段与确认值一致

## 节点化单点运行

支持跳过已完成的步骤，只重跑特定节点：
- **`source-material`**：只重抽源码材料
- **`manual-draft`**：只重写用户手册草稿
- **`application-form`**：只重写申请表
- **`review`**：只复核最终三件套

---

## 推荐的辅助脚本

- `scripts/generate_input_form.py`：生成前置采集表 Markdown
- `scripts/analyze_project.py`：项目结构/语言/路由/文件分析
- `scripts/generate_source_doc.py`：源码材料分页提取 → PDF/DOCX（支持 `--format all`）
- `scripts/generate_source_docx.py`：源码材料直接生成 DOCX（独立使用，支持 `--input` JSON）
- `scripts/generate_manual_draft.py`：用户手册 Markdown 草稿生成
- `scripts/generate_manual_docx.py`：用户手册直接生成 DOCX（含截图嵌入，支持 `--screenshots` 目录）
- `scripts/generate_application_docx.py`：申请表直接生成 DOCX（表格形式）
- `scripts/generate_pdf.py`：Markdown/JSON → 格式化 PDF（支持 source/manual/form 子命令）
- `scripts/final_review.py`：最终三件套复核检查（含 DOCX 文件检查）

## 完成前检查

- 三件套文件存在（每种至少 PDF + DOCX 各一份），文件名包含软件全称和版本号
- 软件全称、版本号、著作权人在三份文档中一致
- 申请表、手册、源码页眉页脚无禁用词或旧项目残留
- 用户手册截图来自真实系统或用户提供文件；暂不截图时有可见预留位
- 代码材料能回溯真实源码，页数、行数、页码满足申报口径
- DOCX 文件可用 Microsoft Word 或 WPS 正常打开编辑
- 已直接检查用户目标目录中的最终文件，而非中间产物
